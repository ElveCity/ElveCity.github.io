# OOP의 특징에 관하여(SOLID principles)
***
   
## SOLID 원칙
   
프로젝트 규모가 커짐에 따라 유지보수, 확장에 문제점이 크게 와닿게 됩니다. 
한 줄 수정에 사이드 이펙트가 발생합니다. 데이터, 로직간의 결합이 강한 탓입니다. 
   
객체지향 프로그래밍은 C언어에서의 절차지향 방식에서는 프로젝트는 갈수록 커지는데 
관리가 안되기 시작해서 나온겁니다. 대규모 프로젝트는 혼자 진행이나 유지가 불가능한데 
해당 프로젝트를 단순히 유지하는것도, 새로운 기능을 넣는것도, 기존 기능을 수정하는 것도 
모두 어렵게 되었습니다. 이를 해결하기 위해 나온 방식이며 객체지향 프로그래밍이 가져야 하는 
주요 특징들을 모아서 정리한 5가지 원칙이 SOLID 원칙입니다. 
   
문제가 발생하는 C언어 코드의 예시는 다음과 같습니다.
   
```C
#include <stdio.h>

typedef enum { ORC, GOBLIN, SLIME } MonsterType;
// 열거형, 몬스터 종류 나열
struct Monster{
	Monster type;
};

// 몬스터 추가 시 매번 함수 Attack 함수 수정 요구, OCP, Open Closed Principle 개방 폐쇄 원칙 위반
// 기존 코드를 매번 건드려야 새 기능 추가 가능
void Attack(struct Monster* m){
	switch(m->type){
		case ORC:
			printf("오크 평타 수행\n");
			break;
		case GOBLIN:
			printf("고블린 평타 수행\n");
			break;
		case SLIME:
			printf("슬라임 평타 수행\n");
			break;
		default:
			printf("환경 데미지");
			break;
	}
}
```
   
SOLID 각 원칙에 대해서 알아보겠습니다. 
   
1) 단일 책임 원칙(SRP, Single Responsibility Principle)
   
하나의 클래스는 하나의 액터에 대해서만만 책임을 가져야 한다는 원칙입니다.
   
클래스 T가 데이터베이스 연결 로직(액터 A의 변경 이유), 데이터 암호화 연산 로직(액터 B의 변경 이유) 
을 동시에 가지고 있다고 할 떄, 데이터베이스의 스키마가 변경되어 클래스의 헤더파일에 새 멤버 변수 혹은 
메서드가 하나라도 추가되는 경우 해당 헤더의 전처리 결과물이 달라지게 됩니다. 
   
전처리 결과물이 달라진다 함은 특정 모듈(클래스)의 바이너리 구조나 인터페이스가 재컴파일되어야 하는 
트리거의 벡터입니다. 벡터는 영향력이 전달되는 경로, 외부의 변경 요구사항이 소스코드에 수정을 요구하게 하는 
유입 경로입니다. 방향성이라고 볼 수도 있겠습니다. 
   
클래스 T는 DB만 변경이 있는건데 암호화까지 같은 헤더에 묶여있어서 암호화 기능만 쓰는 다른 소스 파일까지 
재컴파일되어야 합니다. 빌드시간이 너무 길어지고 의미없는 손해입니다. 시간낭비입니다.

문제가 발생하는 C++코드는 다음과 같습니다.
   
```C++
// Player.h
#pragma once
// 다른 시스템의 헤더를 직접 include, 이로 인해 컴파일 의존성 발생
#include "NetworkSystem.h"
#include "RenderSystem.h"

class Player{

private:
	NetworkSystem net;  
	RenderSystem render;

public:
	void Update();
};
```
   
위를 전방 선언과 포인터 활용으로 해결한 C++ 코드입니다.
   
```C++
// Player.h
#pragma once
// 전방 선언(Forward Declaration)으로 헤더 include 제거
class NetworkSystem;
class RenderSystem;

class Player{

private:
	NetworkSystem* net; // 포인터나 참조를 사용, 크기만 명시
	RenderSystem* render;

public:
	Player();
	~Player();
	void Update();
};
// 실제 include는 Player.cpp 파일 내부에서만 수행, 의존성 격리
```
   
Pimpl(Pointer to Implementation)을 통한 단일 책임 원칙의 구현도 전방 선언과 함께 자주 씁니다. 
   
Pimpl은 포인터의 크기는 항상 고정(64비트 기준 8바이트)되어 있다는 특성을 기반으로 합니다. 
   
클래스의 헤더 파일에는 실제 구현부 대신 8바이트 포인터 하나만 남깁니다. 컴파일러는 Player 클래스의 
전체 메모리 크기를 계산 시 내부 구현이 얼마나 거대한 지 알 필요 없이 포인터 하나 있으니 
8바이트라고 빠른 계산 후 헤더 컴파일이 종료됩니다. 실제 무거운 데이터, 헤더 포함(#include)은 
.cpp 파일 안으로 숨겨집니다. 
   
Pimpl 구현 예시는 다음과 같습니다.
   
```C++
// Player.h (가벼운 헤더)
#pragma once
#include <memory>

class Player{

public:
	Player();
	~Player();
	void Update();

private:
	class Impl; // 전방 선언, Impl 클래스가 존재할 것으로 기대
	std::unique_ptr<Impl> pImpl; // 내부 데이터는 8바이트 포인터로 존재
};

//Player.cpp (무거운 의존성 격리공간)
#include "Player.h"
#include "HeavyDatabase.h" // 헤더가 아닌 cpp에서 include

class Player::Impl{

public:
	HeavyDatabase db; // 멤버 삭제, 추가에 대해서 Player.h는 변화가 없음
	void Update() { dp.Sync(); }
};

Player::Player() : pImpl(std::make_unique<Impl>()) {}
Player::~Player() = default;
void Player::Update() { pImpl->Update(); }
```
   
인터페이스 분리로도 의존성 끊기가 됩니다. 예시는 다음과 같습니다.
   
```C++
// 인터페이스 분리 적용 전, 이동만 필요한 NPC도 공격 함수를 알아야 하는 상황
class ICharacter{

public:
	virtual void Move() = 0;
	virtual void Attack() = 0;
};

// 인터페이스 분리 적용 후, 역할별 분리
class IMovable{
	
pulbic:
	virtual void Move() = 0;
	virtual ~Movable() = default;
};

class IAttackable{

public:
	virtual void Attack() = 0;
	virtual ~IAttackable() = default;
};

// NPC는 자신이 필요한 IMovable만 상속받아 구현, 불필요한 의존성 제거
class NPC : public IMovable{

public:
	void Move override { /* NPC 이동 로직 */ }
};
```
   
위의 Pimpl과 전방 선언의 경우 컴파일 타임의 의존성 분리에 해당합니다. 
#include로 파일과 파일이 텍스트 단위로 엮여 빌드 시간이 길어지는 것을 방지하는 데 주 목적을 둡니다. 
컴파일러쪽에서 볼 때 A파일이 바뀌었는데 B파일도 번역해야하는가? 컴파일 해야하나? 라는 시스템적 
의존성을 끊어냅니다. 물리적으로 의존성을 분리한다고 볼 수 있습니다.
   
논리적 분리인 로직 분리가 있습니다. 
   
먼저 로직 분리의 정의는 다음과 같습니다.
   
- 특정 연산 블록이 접근할 수 있는 메모리 주소 공간(Address Space)의 가시성(Visibility)를 컴파일러 레벨에서 
제한하고, 도메인 간의 상태 변이(State Mutation) 추적을 단방향으로 통제, 최종적으로 런타임의 
명령어 캐시(I-Cache) 지역성을 극대화하는 기법
   
로직 분리가 필요한 경우와 분리를 통해 얻는 이점을 4가지로 추릴 수 있습니다.
   
1) 제어 흐름 그래프(CFG, Control Flow Graph)와 상태 공간(State Space)의 급증 억제
   
거대 단일 클래스(God Class)내에 이동, 전투, 상태 이상 등의 로직이 혼재되는 경우, Update() 함수 
내에 수많은 if, switch문이 중첩되게 됩니다. 이는 컴파일러가 생성하는 제어 흐름 그래프의 
싸이클로매틱 복잡도(Cyclomatic Complexity, 하나의 함수나 모듈 내에서 프로그램 실행 분기점의 총 수+1로 계산) 
를 급증시킵니다.
   
- 분기점 유발 키워드 : if, while , for, case, &&, ||, ? : 등 제어 흐름을 나누는 모든 연산자
   
하나의 클래스가 관리하는 bool 형태의 상태 변수가 N개라고 할 때, 결합된 로직이 가질 수 있는 
이론적 상태 공간의 크기는 2^N개라서 너무 큽니다. 모든 로직이 해당 메모리 공간에 동시 접근이 
가능하므로 런타임 버그 이슈가 발생했을 때 이슈트래킹이 안됩니다.
   
해결책으로써 로직을 두 개의 독립 컴포넌트로 분리하여 각각 N/2개의 변수를 관리하도록 
접근 권한을 단절시키는 방법이 있습니다. 해당 방법의 결과로 상태 공간은 
2*2^(N/2)가 됩니다. 이 정도 크기에서는 괜찮습니다. 
   
유한 상태 기계(FSM, Finite State Machine)의 전이 예측성이 보장됩니다. 
    
FSM은 클라에서 캐릭터 AI나 애니메이션 상태(Idle, Move, Attack, Dead)를 제어하는 주요 아키텍처로 
사용됩니다. 
   
말 그대로 상태 개수가 유한하게 고정되어 있으면서 한 번의 클럭 사이클이나 논리적 틱에서 단 하나의 상태만 
유지 가능하고 특정 입력이나 이벤트에 의해 다른 상태로 넘어가는 전이(Transition)를 모델링한 형태입니다.
   
상태 제어를 FSM으로 모델링 안하고 한다고 할 때 Update()함수 내 if-else가 도배됩니다.
   
상태 제어 FSM 모델링예시는 다음과 같습니다.
   
```C++
class IState{

public:
	virtual void Enter() = 0;
	virtual void Update() = 0;
	virtual void Exit() = 0;
	virtual ~IState() = default;
};

// 각 상태는 독립적인 클래스(로직 분리)로 존재, 다른 상태의 코드를 모르는 형태
class AttackState : public IState{

public:
	void Enter() override { /* 공격 애니메이션 재생 */ }
	void Update() override { /* 데미지 판정 프레임 체크 */ }
	void Exit() override { /* 무기 콜라이더 해제 */ }
};

class Character{
	IState* currentState // 다형성을 활용, VTable로 현재 상태의 함수만 호출

public:
	void ChangeState(IState* newState){
		if (currentState) currentState->Exit();
		currentState = newState;
		if (currentState) currentState->Enter();
	}
	void Tick(){
		if (currentState) currentState->Update(); // 분기문(if/switch) 없이 O(1) 실행
	}
};
```

2) 레지스터 할당(Register Allocation), 스필링(Spilling) 최소화
   
x86-64 아키텍처와 같은 현재 주로 쓰이는 CPU를 기준으로 범용 레지스터(GPR, General Purpose Register)의 개수는 
제한적입니다. 여러 도메인의 로직이 하나의 큰 함수로 묶여 있는 상황이라고 할 때, 연산에 필요한 
지역 변수와 포인터의 개수가 CPU의 레지스터 개수를 초과합니다. 못합니다.
   
컴파일러는 연산할 레지스터가 부족할 시 초과 데이터를 스택 영역으로 내렸다가 다시 올리는 
레지스터 스필링(Register Spilling)을 씁니다. 스택 영역은 메인 메모리쪽이라 레지보다 느립니다. 
로직을 한 함수에 다 구겨넣지 말고 좀 나눠서, 독립된 클래스와 함수로 분리하면 
호출 스택(Call Stack)상에서 실행 컨텍스트(Execute Context)가 뚜렷하게 나뉘어 각 로직 블록이 
실행되는 동안 CPU 레지스터를 온전히 독점하고 최적화해서 쓸 수 있습니다.
   
레지스터가 모자란 예시는 다음과 같습니다.
   
```C++
void UpdatePlayer(float& x, float& y, float vx, float vy, int& hp, int& mp, int buff, float dt){
	// 물리 연산(부동 소수점 레지스터 XMM 및 포인터 주소용 GPR 여러 개 점유)
	x += vx * dt;
	y += vy * dt;

	// 상태 이상 연산(추가적인 범용 레지스터 할당 요구)
	int poisonDmg = (buff > 0) ? 5 : 0;
	hp -= poisonDmg;

	// 전투 연산
	int regen = (mp < 100) ? 2 : 0;
	mp += regen;

	// 이 함수가 끝나기 전까지 수많은 매개변수 포인터와 지역 변수들이 논리적으로
	// 살아있어야 합니다. 한정된 레지스터는 그 시간동안 고갈되며
	// 일부 데이터가 강제로 스택 메모리로 저장되었다가 로드됩니다(Spill/Fill)
}
```
   
다음은 스택으로 보내버렸다가 데려오는 스필링 최소화를 위한 구현입니다.
   
```C++
// 물리 연산 스코프, 연산이 끝나면 물리 관련 레지스터 점유 즉시 반환
void UpdatePhysics(float& x, float& y, float vx, float vy, float dt){
	x += vx * dt;
	y += vy * dt;
}

// 상태 및 전투 스코프, 이전 함수가 반환한 빈 레지스터 재사용
void UpdateStatus(int& hp, int& mp, int buff){
	hp -= (buff > 0 ? 5 : 0);
	mp += (mp < 100 > 2 : 0);
}

// 메인 업데이트 함수, 실행 컨텍스트 분리
void UpdatePlayerOpt(float& x, float& y, float vx, float vy, int& hp, int& mp, int buff, float dt){
	UpdatePhysics(x, y, vx, vy, dt);
	// 함수 호출(Call)이 끝나는 순간 해당 로직의 변수 생명주기가 소멸, 레지스터 확보
	UpdateStatus(hp, mp, buff);
	// 동일한 물리 레지스터 공간을 전투 로직이 덮어씀, 재사용, 빨리됨
}
```
   
다음입니다.
   
3) 명령어 파이프라인(Instruction Pipeline)과 I-Cache(Instruction-Cache) 지역성 높이기
   
너무 큰 로직이 담긴 클래스는 컴파일 시 수천 바이트의 기계어 블록으로 번역되며 해당 코드 실행 시 
CPU의 L1 명령어 캐시 용량 초과가 뜨면 지속적으로 캐시가 축출됩니다(Cache Eviction).
   
CPU가 메모리에서 명령어를 가져오느라 파이프라인이 멈추게 되며 이게 스톨(Stall) 현상입니다.
   
로직 분리를 통해 도메인 별로 클래스를 나눕니다. 이러면 특정 도메인 연산 수행 시 
해당 모듈의 작고 응집도 높은 기계어 블록만 L1 명령어 캐시에 드갑니다. 
이를 통해 분기 예측(Branch Prediction)의 성공률이 높아지며 이는 CPU 사이클 낭비가 없어지는 
결과를 냅니다. 좋습니다.
   
명령어 캐시 미스 발생 예시입니다. 다형성으로 명령어 캐시가 오염된 사례입니다.
   
```C++
class IEntity { public: virtual void Update() = 0; };
class Player : public IEntity { void Update() override { /* 아주 많은 물리 로직 */ } };
class Monster : public IEntity { void Update() = override { /* 너무 많은 AI 로직 */ } };

void UpdateAll(std::vector<IEntity*>& entities){
	for (auto* entity : entities){
		// Player 기계어 로드 -> 파이프라인 플러시 -> Monster 기계어 로드 (반복)
		// 객체가 바뀔 때마다 전혀 다른 기계어가 I-Cache를 덮어 씀, 캐시 미스 아주아주높아짐
		entity->Update();
	}
}
```
   
도메인 로직 분리, 일괄 처리로 명령어 지역성을 높여 해결한 케이스입니다.
   
```C++
// 물리 로직만 담당(해당 함수의 기계어 블록은 작고 응집됨)
void UpdatePhysics(std::vector<PhysicsData>& physics){
	// 루프가 도는 내내 물리 연산 기계어사 L1 명령어 캐시에 상주
	// 조건 분기가 적어서 파이프라인 분기 예측(Branch Prediction)이 잘됨
	for (auto& p : physics) { p.x += p.vx; }
}

// AI 로직만 담당
void UpdateAI(std::vector<AIData>& ais){
	// 앞에 루프가 다 끝나고 AI 연산 기계어만 명령어 캐시에 드감, 반복 실행
	for (auto& ai : ais) { ai.targetId = 1; }
}

void ProcessFrame(std::vector<PhysicsData>& p, std::vector<AIData& a>){
	// 로직 실행 흐름을 섞지 않고 도메인 단위로 단절
	UpdatePhysics(p);
	UpdateAI(a);
}
```
   
다음입니다. 
   
4) 포인터 에일리어싱(Pointer Aliasing) 방지, 링킹 격리
   
포인터 에일리어싱은 서로 다른 이름의 포인터나 참조자 변수 2개 이상이 물리적으로 
동일한 메모리 주소를 가리키는 상태입니다. 메모리가 겹친겁니다.
   
컴파일러는 포인터 에일리어싱의 가능성이 냐있는 코드에 대해서 명령어 재배치(Instruction Scheduling)나 
벡터화(SIMD 명령어 사용, Single Instruction, Multiple Data, 여러 데이터를 묶어 처리하는 벡터 연산 변환 최적화) 같은 
고급 컴파일러 최적화를 하지 않게 됩니다. 
   
로직 분리, 각 모듈이 완전하게 다른 메모리 주소 체계를 가지도록 강제하며 인터페이스의 매개변수를 
통해서만 값을 복사(Pass by Value)해서 전달 시, 컴파일러는 두 모듈의 메모리가 겹치지 않음을 
증명할 수 있기에 좋은 최적화의 수행이 가능합니다.
   
예시는 다음과 같습니다.
   
```C++
// pHp와 pDamage가 같은 주소일 가능성 존재, 컴파일러가 매번 메모리(RAM)에서 값을 다시 읽어옴
void ApplyDamage(int* php, int* pDamage, int count);

// __restrict 사용, 메모리가 독립적임을 보증, 최적화 진행
void ApplyDamageOpt(int* __restrict pHp, const int* __restrict pDamage, int count){
	// 컴파일러는 두 배열이 겹치지 않음을 확인, -03 최적화 진행 가능
	// 해당 최적화에서 이 루프를 128/256비트 SIMD(AVX) 명령어로 전체 벡터화
	// 한 클럭에 여러 개의 데이터 동시 연산 진행
	for (int i = 0; i < count; ++i){
		pHp[i] -= pDamage[i];
	}
}
```
   
private 사용은 문법적인 접근만 막습니다. 
   
컴파일된 바이너리 심볼 테이블에는 이름이 남고 다른 파일과 링킹 충돌 발생이 일어날 수 있습니다. 
   
해결 방법은 익명 네임스페이스(Anonymous Namespace) 사용입니다. 
이는 현재 번역 단위(.cpp 파일)내부로 온전히 격리시키는 기법입니다.
   
예시는 다음과 같습니다.
   
```C++
// CombatSystem.cpp 파일 내부
// 익명 네임 스페이스, 해당 블록 내의 모든 데이터 및 함수는 내부 링키지를 가짐
// 이 .cpp 파일을 벗어난 외부 시스템에서는 존재 파악 불가능, 링킹 차단

namespace{
	const float CRIT_MULTIPLIER = 1.5f; // 다른 파일에 같은 이름의 변수가 있어도 충돌 안함

	// 이 파일에서만 쓰이는 내부 연산 로직을 헤더 파일에서 은닉
	float CalculateBaseStats(float baseAtk){
		return baseAtk * 1.2f;
	}
}

void CombatSystem::ExecuteAttack(float atk){
	// 링커 레벨에서 완전히 격리된 내부 심볼을 호출
	float finalDmg = CalculateBaseStats(atk) * CRIT_MULTIPLIER;
}
```
   
다음으로는 컴파일러 레벨별 최적화, 단일 책임 원칙과 관련된 ECS 아키텍처에 대해 
알아보겠습니다.
