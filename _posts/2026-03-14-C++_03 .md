# OOP의 특징에 관하여
***
   
## 클래스, 객체(Class, Object)
   
기존 C언어에서는 상태(Data)와 행위(Function)간 관계성이 소스 코드 상 표현이 없었습니다. 데이터 처리를 위해서 
매번 구조체의 포인터를 함수의 매개변수로 명시적으로 넘겨주었고, 구조체의 생성, 소멸 또한 수동으로 init(), destroy() 
를 호출하여 관리하였습니다. 
   
C++에서의 클래스(Class) 개념은 데이터와 함수를 하나의 스코프(Scope)로 묶어내는 형태입니다. 
클래스는 객체를 생성하기 위한 일종의 청사진(Blueprint)에 해당하며 이 청사진을 바탕으로 
메모리에 실체화 할 때 생성되는 것이 객체(Object or Instance)입니다.
   
## this pointer, RAII pattern
   
C언어에서 명시적으로 구조체 포인터를 넘기던 방식은 this 포인터가 그 역할을 수행하게 됩니다. 
C++에서 클래스의 멤버 함수 호출 시, 컴파일러 단에서 객체 자신의 메모리 주소를 가리키는 this 포인터가 
함수의 첫 번째 인자로 전달되는 형태입니다. 
   
빈번히 발생되던 메모리 누수(Memory Leak)와 자원 관리의 복잡성의 해결을 위해서 
RAII(Resource Acquisition Is Initializaion) 개념이 도입됩니다. 
   
이는 생성자(Constructor)와 소멸자(Destructor) 멤버 함수를 통해, 메모리 내 객체 생성 시 
필요 자원(메모리 할당, 파일 핸들, 스레드 락 등)을 자동으로 흭득하고, 객체가 스코프를 벗어나 
소멸 시 컴파일러가 소멸자를 호출하여 자원을 무조건적으로 반환하도록 보장하는 형태입니다. 
   
클래스, 생성자, 소멸자, 멤버함수, 객체화의 예시는 다음과 같습니다. 
   
```C++
#include <iostream>
#include <string>
		// 데이터, 행위, 생명주기 관리가 하나로 합쳐진 클래스
class Player{
private: 
	std::string name;
	int hp;

public:
		// 생성자(Constructor): 객체 생성 시 자동 호출, 초기화 보장
	Player(const std::string& playerName, int initialHp)
		: name(playerName), hp(initialHp){
		std::cout << name << "플레이어가 생성되었습니다." << "\n";
		}

		// 소멸자(Destructor): 객체 소멸 시 자동 호출, 자원 해제 보장
	~Player(){
		std::cout << name << "플레이어가 소멸되었습니다." << "\n";
	}

		// 멤버 함수: this 포인터가 암시적으로 전달되므로 객체 상태에 직접 접근 가능
	void TakeDamage(int damage){
		hp -= damage;
		std::cout << name << "의 남은 체력: " << hp << "\n";
	}
};

int main(){
	{
			// 객체화 (Instantiation): 스코프 진입 시 생성자 호출
		Player p1("Warrior", 100);
		p1.TakeDamage(20);
	}		// 스코프를 벗어나는 순간, 컴파일러가 알아서 소멸자를 호출, 안전하게 정리

	return 0;
}
```
   
위 코드의 메인 함수에서는 중괄호를 한 번 더 사용하여 return 0 전 소멸자를 호출합니다. 
이는 생명주기를 앞당겨 통제한 상황입니다. 큰 프로젝트에서 시스템의 성능, 안정성을 고려하는 방식의 기초적인 형태입니다. 
   
다음의 두 가지 상황을 볼 수 있습니다. 
   
1) 임시 메모리(메가바이트 단위)의 즉각적 해제
   
저 객체가 100바이트짜리 플레이어가 아닌 초당 60번씩 생성, 소멸되어야 하는 50MB 임시 텍스처 버퍼라고 
가정되는 경우 함수가 끝나기를 기다렸다 메모리 해제 시 함수 실행 도중 메모리 사용량 문제로 운영체제가 프로그램을 
강제로 종료(OOM, Out Of Memory)하는 상황이 발생할 수 있습니다. 
중괄호로 묶음으로써, 객체의 사용이 끝나는 즉시 메모리를 운영체제에 반환하여 메모리 피크(Peak)를 낮출 수 있습니다. 
   
2) 스레드 동기화와 임계 구역(Critical Section)의 자동 해제(Lock Guard)
   
서버 혹은 멀티스레드 클라이언트에서는 여러 스레드가 동시에 자원에 접근하는 것을 막기 위해 락(Lock)을 겁니다. 
이 락의 예시는 다음과 같습니다. 

```C++
void UpdateGameWorld(){
	{
	std::lock_guard<std::mutex> lock(myMutex);	// 스코프 진입 시 자동 잠금
		// 몬스터 체력 계싼 등 민감 데이터 수정
	}	// 스코프를 빠져나가는 순간, 컴파일러가 알아서 소멸자 호출 및 잠금 해제

		// 여기부터 다른 스레드가 다시 접근 가능 (무거운 파일 I/O 등 수행)
}
```
   
위와 같은 케이스에서 함수 내의 추가적인 {} 없이 함수 끝까지 락이 유지되는 경우 게임 전체의 프레임 드랍이 발생할 수 있습니다. 
   
## .과 ->의 차이
   
- . 직접 멤버 접근 연산자(Dot Operator)
   
직접 멤버 접근 연산자는 메모리상의 객체 그 자체(Value)의 멤버에 직접 접근합니다. 
   
- -> 간접 멤버 접근 연산자(Arrow Operator) 
   
간접 멤버 접근 연산자는 객체가 있는 곳의 주소를 담은 포인터(Pointer)를 통해 멤버에 접근합니다.
   
클래스, 구조체에서는 여러 타입의 데이터를 메모리에 연속적으로 나열하는 형태를 가집니다. 
컴파일러는 소스 코드 빌드 시, 클래스 내부의 각 멤버가 객체의 시작 주소(Base Address)로부터 
몇 바이트 떨어져 있는지(Offset)를 미리 계산, 고정합니다. 

```C++
struct Vector2{
	int x;
	int y;
};
```
   
위와 같은 구조체에서 x에 접근하는 경우, y에 접근하는 경우 모두 최종 연산은 시작 주소를 찾은 뒤, 
정해진 오프셋을 더해서 메모리에 읽고 쓰는 형태가 됩니다. 
   
- 어셈블리 레벨 해석 차이
   
```C++
void TestAccess{
	Vector2 obj;	// 스택에 객체 직접 생성
	Vector2* ptr = &obj;	// 포인터에 obj 주소 저장

	obj.y = 10; // . 연산자를 사용한 접근

	ptr->y = 20; // -> 연산자를 사용한 접근
}
```
   
위의 코드를 기준으로 두 접근간의 차이에 대해서 확인해보겠습니다. 
   
> obj.y = 10; (. 연산자 사용 시 어셈블리 해석)

```   
mov		DWORD PTR [rbp-4], 10
```
   
. 연산자의 경우 컴파일 타임에 시작 주소가 스택 포인터(rbp or rsp) 기준의 상대 주소로 확정되어 있습니다. 
   
컴파일러는 obj가 스택 메모리 [rbp-8]에 위치함을 인지하고 있으며 y의 오프셋은 +4이기에 최종 주소는 [rbp-4]가 됩니다. 
CPU는 레지스터를 거치거나 메모리에서 주소를 따로 읽어올 필요 없이 계산된 타겟 메모리 주소([rbp-4])에 
곧바로 10의 상수를 넣습니다. 메모리 접근은 1회(쓰기) 발생합니다. 
   
> ptr->y = 20; (-> 연산자 사용 시 어셈블리 해석)
   
-> 연산자의 경우 컴파일러는 ptr이 가리키는 실제 객체가 힙(Heap)에 있는지, 스택에 있는지 모를 수 있습니다. 
ptr이라는 변수 내에 주소값이 들어있다는 사실만 인지하고 있습니다. 
   
ptr 변수 자체는 스택 [rbp-16]에 저장되어 있습니다.
   
따라서 CPU는 먼저 ptr 변수가 위치한 메모리([rbp-16])에 접근하여 그 안의 주소값을 범용 레지스터(rax)로 
읽어오는(Load) 작업을 추가로 수행해야 합니다. 이는 포인터의 역참조(Dereference) 과정에 해당합니다. 
   
이는 다음과 같습니다.
```
	mov		DWORD PTR [rax+4], 20
```
   
그 후, 레지스터 rax에 담긴 주소를 베이스삼아 오프셋 +4를 더해 타겟 메모리에 값(20)을 씁니다. 
메모리 접근은 총 2회(읽기 1회, 쓰기 1회) 발생합니다.
   
이는 다음과 같습니다.
```
	mov		DWORD PTR [rax+4], 20
```
   
-> 연산자는 메모리 간접 참조(Memory Indirection)을 유발합니다. 
   
ptr->y를 수행하기 위해서 ptr의 값을 읽어오는 경우, 그 주소가 CPU 캐시(Cashe)에 없다면 캐시 미스(Cashe Miss) 
가 발생하여 수백 클럭 사이클 손해가 발생할 수 있습니다. 
   
클라이언트 혹은 시스템 아키텍처에서 그럼에도 ->의 사용이 빈번한 이유로 함수의 인자로 객체 전체를 
복사(. 의 사용)보다 포인터의 사용으로 주소만 넘겨주는 방식의 간접 참조(-> 의 사용)를 사용하는 편이 낭비를 크게 줄이기 때문입니다. 
수십 수백메가의 몬스터 객체나 텍스처 데이터를 함수로 인자를 넘긴다 할 때 .는 통으로 객체를 복사하고 
-> 는 8바이트 포인터 하나만 넘기고 맙니다.
