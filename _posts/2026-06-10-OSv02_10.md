# Operating System_10
***
   
## Threads Safe. 스레드 안전
   
자료구조에 락을 추가해서 여러 스레드가 동시에 접근하더라도 데이터의 일관성이 
께지지 않고 올바르게 동작하는 상태를 의미.
   
## Concurrent Counter. 병행 카운터
   
가장 단순한 형태의 스레드 안전 자료구조.
   
기본적으로 단일 전역 락을 사용하여 구현하지만, 스레드 수가 늘어날수록 
성능이 급격히 저하되는 확장성 문제를 가짐.
   
## Approximate Counter. 근사 카운터 / Sloppy Counter
   
병행 카운터의 확장성 문제를 해결하기 위한 기법.
   
각 CPU 코어마다 지역 카운터(Local Counter)와 지역 락을 두어 평소에는 
경합 없이 숫자를 셈.
   
특정 임계값(Threshold)에 도달했을 때만 전역 락을 잡고 전역 카운터(Global Counter)를 업데이트함.
   
## Concurrent Linked List. 병행 연결 리스트
   
리스트 작업(삽입, 검색 등) 시 전역 락을 사용하여 보호함.
   
검색 작업 시 임계 영역을 일찍 벗어나도록 코드를 재작성해서 병행성을 약간 높일 수 있음.
   
## Hand-over-hand locking(Lock Coupling)
   
리스트의 병행성을 극대화하기 위해서 리스트 전체가 아닌 노드마다 락을 두는 기법.
   
리스트를 탐색할 때 다음 노드의 락을 흭득한 후에만 현재 노드의 락을 해제하며 전진함.
   
이론적으로는 병행성이 높지만, 매 노드마다 락을 잡고 푸는 
오버헤드가 커서 실 성능은 떨어질 수 있음.
   
## Concurrent Queue. 병행 큐
   
Head lock와 Tail lock을 분리해서 Enqueue와 Dequeue 작업이 동시에 일어날 수 있도록 함.
   
이 과정에서 두 작업을 완전히 분리하기 위해서 Dummy Node를 사용함.
   
## Concurrent Hash Table. 병행 해시 테이블
   
단일 전역 락 대신, 해시 버킷(Bucket)마다 개별 연결 리스트와 락을 두어 구현함.
   
이를 통해 서로 다른 버킷에 접근하는 스레드들은 동시에 작업을 수행할 수 있어 
아주 좋은 성능과 확장성을 보여줌.
   
## Premature Optimization. 조기 최적화 / Knuth's Law
   
처음부터 복잡한 락킹 기법을 쓰지 말고, 가장 단순한 방식(단일 전역 락)으로 
먼저 구현한 뒤 성능 문제가 실제로 발생했을 경우에만 최적화를 진행하라는 설계 원칙.
