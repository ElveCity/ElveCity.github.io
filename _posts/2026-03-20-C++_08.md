# OOP의 특징에 관하여(Interface, Virtual, Abstraction)
***
   
## 인터페이스 활용을 통한 다중 상속(상속받은 것에 대한 상속) 데이터 중복 제거 및 안정성 구현
   
C언어에서는 상속이 존재하지 않기에 부모 역할의 구조체를 자식 구조체의 첫 번째 멤버로 선언(동일한 주소의 메모리 구조 이용) 
하게 됩니다. 다중 상속과 유사한 형태의 구현을 위해서 여러 개의 구조체를 포함시키는 경우 메모리 레이아웃이 복잡해지며 
데이터의 물리적 중복 할당이 발생하게 됩니다. 
   
데이터 물리적 중복 할당으로 인하여 발생되는 문제는 크게 3가지로 볼 수 있습니다.

1) 동일 의미 데이터가 서로 다른 위치에 두 개 존재함으로써 메모리 오염 및 낭비가 발생합니다.
   
2) 한 쪽 부모를 통해 데이터 a를 수정해도 다른 쪽 부모의 a는 변경되지 않아 객체 상태가 모순적이게 되어 
상태 불일치(Inconsistency)가 발생합니다.
   
3) 컴파일러는 자식 객체에서 데이터 a에 접근 시, 어느 쪽 부모의 데이터 참조가 필요한지 
판단이 불가능하여 컴파일 에러가 발생합니다.
   
다중 상속된 형태를 구현 후 특정 부모 멤버 접근 시, 구조체의 메모리 시작 주소에서 해당 데이터 위치까지 오프셋을 
직접 계산, 포인터 캐스팅이 요구되었습니다. 휴먼에러가 발생하기 쉬운 부분이고 런타임 메모리 침범(Segmentation Fault) 등의 
버그가 발생할 수 있었습니다. 
  
C언어의 다중상속 구현은 다음과 같습니다.
   
```C
#include <stdio.h>

// 최상위 상태 구조체
typedef struct {int health;} BaseState;

// BaseState 포함 두 구조체(다중 상속 중간 단계)
typedef struct {BaseState base; int attack;} Derived1;
typedef struct {BaseState base; int defence;} Derived2;

// 다중 상속 구현 구조체
typedef struct{
	Derived1 d1;
	Derived2 d2;
} DiamondObject;

int main(){
	
	DiamondObject obj;

	// 하나의 객체 안에 health가 두 개 존재하게 됨으로써 데이터 중복, 모호성 발생
	obj.d1.base.health = 100;
	obj.d2.base.health = 50;

	// 어느 쪽 health가 진짜 객체의 데이터인지 확인 불가능, 관리도 불가능
	printf("D1 Health : %d, D2 Health: %d\n", obj.d1.base.health, obj.d2.base.haelth);
	
	return 0;
}
```
   
이렇게 발생하는 문제(데이터 중복, 구현 문제, 휴먼 에러 발생 빈도 높음..)의 해결을 위해 
C++로 넘어오면서 순수 가상 함수와 인터페이스가 도입되었습니다. 
   
이를 통해 언어 차원에서 다중 상속을 지원하게 되었으며 컴파일러는 메모리 레이아웃, 포인터 오프셋 계산을 
자동으로 처리하게 됩니다. 수동 포인터 캐스팅이 없어졌기에 이 부분에서의 휴먼에러 빈도를 크게 낮출 수 있게 됩니다. 
   
virtual 키워드를 통한 가상 상속(Virtual Inheritance)이 C++에서 도입되었습니다. 
가상 상속에서는 가상 기본 클래스 테이블(vbtable)메커니즘이 제공되는데 이 테이블을 통해서 
최상위 부모의 데이터를 메모리 상에 단 하나만 존재하도록 보장할 수 있습니다. 
   
멤버 변수 없이 순수 가상 함수만으로 이루어진 클래스를 정의함으로 인터페이스 구현이 가능하게 됩니다. 

- 순수 가상 함수(Pure Virtual Function)
   
순수 가상 함수 사용 시 부모 클래스에서 미구현, 상속받는 자식 클래스에서 필히 해당 함수를 구현해야 함이 강제됩니다. 
   
가상 함수 선언부 끝에 = 0;을 붙이는 방식으로 사용하며
순수 가상 함수가 하나라도 클래스에 포함되는 경우 해당 클래스는 추상 클래스(Abstract Class)로 
취급됩니다. 추상 클래스는 불완전하며 직접 객체로 생성(인스턴스화)할 수 없습니다. 
   
추상 클래스는 부모 클래스에서 공통된 호출 방식인 인터페이스만 제공하고 실제 동작(ex. 상태 변화, 렌더링, 타격 판정 등) 
은 자식 클래스의 특성에 맞게 위임하기 위해 사용합니다.
   
- virtual 키워드
   
virtual 키워드를 적용하는 경우 런타임, 프로그램 실행 시점에서 객체를 보고 바인딩 할지 말지 봅니다. 
   
일반적인 virtual이 없는 함수는 컴파일 타임에 정적 바인딩(Static Binding)이 일어납니다. 
   
예시로 enemy->Attack() 이 코드가 IEnemy::Attack 함수의 고정된 메모리 주소(ex. 0x1000)로 점프 해라 
(ex. CALL 0x1000) 기계어 주소 맵핑
   
virtual 키워드가 들어가는 경우 주소 맵핑이 컴파일 타임이 아닌 런타임으로 밀립니다. 

virtual이 들어가는 경우의 동작 과정은 위의 예시(enemy->Attack())를 기준으로 다음의 두 단계로 볼 수 있습니다.
   
1) 컴파일 타임(준비)
   
컴파일러는 virtual 함수를 가진 클래스를 위해 가상 함수 테이블(vtable)이라는 배열을 데이터 영역 메모리에 만듭니다. 
이 배열에는 해당 클래스가 실행해야 하는 진짜 함수들의 메모리 주소가 순서대로 들어갑니다.
   
이후 객체가 생성될 때 객체의 메모리 맨 앞단에 이 vtable 주소를 가리키는 4바이트 혹은 8바이트 포인터, 
가상 포인터(vptr)를 삽입합니다. 이로 인해 객체의 메모리가 커집니다.
   
2) 런타임 (결정, 실행)
   
프로그램 실행 중 enemy->Attack()를 만나게 되는 경우 CPU는 고정된 주소로 바로 점프하지 않으며 
현재 enemy 포인터가 가리키고 있는 힙 메모리의 실제 객체로 찾아가서 객체의 메모리 맨 앞의 vptr를 읽습니다. 
vptr이 가리키는 vtable 메모리 영역으로 가서 Attack()함수가 있는 인덱스에 있는 진짜 메모리 주소(ex. 해당 몬스터의 Attack 주소 0x2050)를 
꺼내옵니다.
   
마지막으로 CPU의 명령어 포인터(Instruction Pointer)를 그 주소(0x2050)로 변경하여 코드를 실행합니다. 
   
가상 함수는 동적 바인딩을 진행합니다. 가상 함수의 동적 바인딩은 다형성의 실현을 위해, 부모 타입의 포인터나 
참조자로 오버라이딩된 함수를 호출 시, 객체 내부에 삽입된 가상 포인터 vptr를 참조, 가상 함수 테이블 vtable에서 
실제 객체가 가진 함수의 메모리 주소를 읽어와 최종 실행 코드를 찾는 일련의 과정입니다.
