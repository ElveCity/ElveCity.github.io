# OOP의 특징에 관하여(상속성)
***
   
## 상속성(Inheritance)
   
C언어에서는 상속(Inheritance)이 존재하지 않았으며 데이터를 묶는 수단은 구조체만 존재하였습니다. 
   
프로젝트 규모가 커지면서 비슷한 데이터를 가진 구조체를 매번 새로 정의하지 않고 재사용하는 방법에 대한 
고민으로 나오게 된 것이 상속입니다. 
   
C언어에서는 구조체 사용 시 메모리 구조 상 첫 번째 멤버의 주소가 전체 구조체의 주소와 같다를 기반으로 
구조체 안에 다른 구조체를 첫 번째 멤버로 포함시키는 방식(메모리 레이아웃 트릭)으로 상속과 비슷한 형태를 만들어 사용하였습니다. 
이 경우 매번 강제 형변환(Casting)을 수행해야하고 불안전했으며, 함수를 상속받기 위해서는 복잡한 
함수 포인터를 구조체마다 일일히 연결해야 했습니다. 
   
언어 차원에서 컴파일러의 메모리 정렬, 가상 함수 테이블을 통한 함수 포인터 관리를 지원하는 것을 포함한 것이 
C with Classes, C++입니다. 상속은 이런 형태로 시작되었습니다. 
   
C언어에서의 상속과 비슷한 형태의 구현은 다음과 같습니다.
```C
#include <stdio.h>

	// 기반 역할을 할 구조체의 정의
typedef struct{
	int id;
	float x, y;
} GameObject;

	// 파생 역할을 할 구조체의 정의
typedef struct{
	GameObject bse;		// 반드시 첫 번째 멤버로 배치해야 전체 구조체 주소와 같아서 메모리 정렬이 가능함
	int hp;
} Player;

	// 기반 객체를 다루는 공통 함수
void Move(GameObject* obj, float dx, float dy){
	obj->x += dx;
	obj->y += dy;
	printf("ID %d 이동 : (%.1f, %.1f)\n", obj->id, obj->x, obj->y);
}

int main(){
	Player p1;
	p1.base.id = 1001;		// base를 거쳐야 하기에 접근이 번거로움
	p1.base.x = 0.0f;
	p1.base.y = 0.0f;
	p1.hp = 100;

		// Player 포인터를 GameObject 포인터로 강제 형변환하여 전달(강제 형변환으로 인한 위험성 존재)
	Move((GameObject*)&p1, 5.0f, 3.0f);

	return 0;
}
```
   
상속이 들어간 C++에서의 구현은 다음과 같습니다.
```C++
	// 부모 클래스 정의
class GameObject{

protected:	// 자식에게만 접근 허용
	int id;
	float x, y;

public:
	GameObject(int _id) : id(_id), x(0.0f), y(0.0f) {}

	virtual void Move(float dx, float dy){
		x += dx;
		y += dy;
		std::cout << "ID " << id << " 이동 : (" << x << ", " << y << ")\n";
	}
	virtual ~GameObject() = default;
};

	// 자식 클래스 정의(C++ 상속 사용)
class Player : public GameObject{

private:
	int hp;

public:
		// 부모의 생성자를 명시적으로 호출
	Player(int _id, int _hp) : GameObject(_id), hp(_hp) {}

		// 부모의 데이터를 base 키워드 없이 직접 접근 가능
	void TakeDamage(int damage){
		hp -= damage;
		std::cout << "ID " << id << " 체력이 감소되었습니다. 남은 체력 : " << hp << "\n";
	}
};

int main(){
	Player p1(1001, 100);

		// 강제 형변환 없이 부모의 함수 호출
	p1.Move(5.0f, 3.0f);
	p1.TakeDamage(10);

	return 0;
}
```
   
상속의 사용 이유는 프로젝트 복잡성 관리 때문입니다. 
장점은 크게 3가지로 볼 수 있겠습니다. 
   
1) 코드의 재사용성, 확장성
   
부모 클래스에 공통적인 속성(ex. 좌표, ID 등)과 행위(ex. 이동)를 정의해둘 때, 자식 클래스들은 이를 
그대로 물려받아 그에 더해 해당 클래스의 특수 기능(ex. 스킬 사용, 몬스터 추적 AI 등)만 추가적으로 구현하면 됩니다. 
코드 중복을 줄일 수 있습니다. 
   
2) 계층화(Hierarchy)를 통한 모델링
   
분류(Classification) 개념을 코드에 도입할 수 있습니다. 
포유류 -> 고양이 처럼 코드를 논리적인 계층으로 나누는 것이 가능합니다. 
인간이 보기에 좀 더 직관적인 구성을 가능케합니다. 
   
3) 다형성(Polymorphism)의 기반
   
모든 몬스터(슬라임, 버섯, 나무)는 결국 몬스터(부모)다. 라는 논리를 통해 몬스터 포인터 배열 하나만으로 
수천 마리의 다양한 몬스터를 한 번에 관리하고 업데이트 할 수 있습니다. 중요합니다. 
   
이러한 상속의 사용이 과도한 커플링을 만드는 게 아닌가 하는 생각을 할 수 있습니다. 
상속은 객체지향 패러다임에서 클래스 간 맺을 수 있는 강력한 결합(Tight Coupling)입니다. 
이 때문에 발생할 수 있는 문제는 깨지기 쉬운 기반 클래스(Fragile Base Class)가 있습니다. 
   
해당 문제는 부모 클래스 내부 로직을 약간 수정하거나 멤버 변수를 하나 추가하는 경우 이를 상속받은 
수십, 수백의 자식 클래스들이 연쇄적으로 영향을 받아 예상하지 못한 버그를 발생하는 경우입니다. 
   
이 때문에 프로젝트 기획 변경으로 여러 계층에 특정 기능 도입 시 단일 상속 구조로 구성되는 경우 
부모 클래스에 넣기도, 모든 자식 클래스에 따로 구현하기도 참 애매해집니다. 상속은 수직적 확장을 지향하며 
결합도가 쎈 만큼 변경이 어렵습니다. 
   
이론적으로 접근할 때 상속에서는 두 가지를 크게 봅니다. 
   
1) is-a 관계(Subtyping)가 성립하는가
   
상속은 논리적으로 자식 클래스는 부모 클래스의 일종(Derived is a Base)이 참인 경우에만 사용해야 합니다. 
Player is a Character, Monstser is a Character 둘 다 성립되니까 상속을 사용해도 별 문제가 없습니다. 
단순히 코드를 재사용하기 위해서 논리적 관계가 없는 클래스를 상속하고 받고 이런건 객체지향 설계 원칙 위배로 봅니다. 
   
2) 생성자와 소멸자의 호출 순서(생명 주기)
   
객체가 메모리에 할당될 때, 기반 클래스의 생성자가 먼저 호출되고 파생 클래스의 생성자가 호출됩니다. 
객체가 소멸할 때는 파생 클래스의 소멸자가 먼저 호출된 후 기반 클래스의 소멸자가 호출됩니다. 
부모 클래스를 위, 상대적으로 자식 클래스를 아래쪽에 있다고 가정하는 경우 생성자는 위에서 아래로, 
소멸자는 아래에서 위로 호출되는 형태입니다. 뭐 그렇습니다. 
   
C++ 기반 구현에서는 상속 시에도 public, protected, private 접근 지정자를 쓸 수 있습니다. 
다형성을 위한 is-a 관계 모델링 시 public 상속을 사용해야 하며 protected 상태는 외부(main 함수 포함)에서 접근할 수 
없지만 상속받은 자식 클래스 내부에서는 접근할 수 있도록 허용하는 형태입니다.
   
상속은 무분별하게 사용 시 코드를 경직되게 만들기에 보수적으로 사용해야 합니다. 
   
따라서 상속(Inheritance, is-a 관계)보다는 합성(Composition, has-a 관계)의 구현이 주가 됩니다. 
이를 통해서 구현의 재사용을 높입니다.
   
상속 목적 기반 클래스의 소멸자에는 virtual을 꼭 붙여서 가상 소멸자(Virtual Destructor)로 구현해야 합니다. 
다형성을 위해 부모 클래스의 포인터로 자식 객체 관리 시, 부모 클래스의 소멸자가 가상 소멸자로 선언되어있지 않은 경우 
자식 클래스의 소멸자가 호출되지 않으며 메모리 누수(Memory Leak)가 발생하게 됩니다. 
   
상속을 사용하지 않거나(클래스 레벨) 상속 기반으로 존재하는 오버라이딩을 쓰지 않을 때(함수 레벨) 
final을 사용하여 구현 의도를 명확히 할 수 있습니다. override 사용 시 부모 가상 함수 재정의로 구현하다가 
오타내서 새 함수가 나오는 그런 휴먼 에러를 컴파일 타임에 잡을 수 있습니다. 
   
일반 상속은 꽤 위험한 방식입니다. 
   
구현(상태와 로직)을 그대로 물려주는 형태가 일반 상속인데 부모클래스쪽에서 내부 멤버 변수 하나가 
수정된다거나 함수 로직이 바뀐다거나 이런 일이 발생했을 때 의존하고 있는 전체 파일을 다시 컴파일(Rebuild) 해야합니다. 
개발 생산성이 처참해집니다. 
   
이런 문제를 해결하기 위해 구현이 전혀 없는 인터페이스 상속이 나오게 되었습니다. 
   
C++에서 인터페이스는 순수 가상 함수만 존재하며 멤버 변수가 전혀 없는 추상 클래스입니다. 
타 클래스에서는 추상 클래스의 인터페이스 포인터만 들고 있는 식으로 구현됩니다. 
   
인터페이스 포인터만 들고 있는 클래스를 GameLogic 클래스라고 하고 추상클래스를 IRenderer라 할 때 
GameLogic은 실제 렌더러가 내부적으로 어떻게 구현되어있는지 딱히 알 필요가 없는 형태입니다. 
헤더 파일을 포함할 필요도 없고 실제 렌더러를 DirectX11Renderer라 할 때 그 내부 코드가 어떻게 고쳐지든 
인터페이스의 형태. 순수 가상 함수를 (virtual void Draw() = 0;)라 할 때, Draw()라는 함수 이름과 매개변수 
만 바뀌지 않는다면 GameLogic은 재컴파일 할 일이 없습니다. 
이를 물리적 파일 의존성을 끊어내는 Decoupling이라고 볼 수 있습니다. 
이런 식으로 인터페이스 상속은 모듈 간 커플링(Coupling, 물리적/논리적 결합도에 해당)을 낮추고 끊어내고 그런걸 합니다.
