# ECS Architecture, Entity Component System Architecture
***
   
## ECS Architecture 도입 배경
   
객체 지향 프로그래밍에서는 기능 확장을 위해 상속을 사용하였으며 그 상속이 중첩되어 다중 상속이 
빈번하게 사용되었습니다. 상속이 많아지고 깊은 상속트리가 구현됨에 따라서 커플링이 쎄지고 
가상 함수 테이블(vtable)의 참조 또한 오버헤드가 크게 발생하게 됩니다. 
   
이 시스템에서는 객체 할당에 있어서 힙 메모리 영역에 산발적으로 할당되며 L1/L2 캐시 활용도가 낮아지고 
캐시미스가 잦아집니다. 산발적으로 되는 만큼 중간중간 공간부족이 생겨가지고 메모리 단편화도 발생합니다.
   
ECS로 가기 전 CBA(Component Based Architecture)가 그 중간단계로 들어가게 됩니다.
   
CBA에 와서 기존의 Is-A 관계에서 Has-A 관계로 전환이 발생합니다. 
   
이는 수직적 상속(Inheritance)관계에서 수평적 합성(Composition)관계로의 전환과 같습니다. 
   
수평적으로 관계가 변하면서 자식 클래스에 대한 리플 이펙트 또한 해소됩니다. 
   
각 기능을 독립 클래스인 컴포넌트로 분리하고 이를 런타임에 빈 컨테이너에 Attach하는 식으로 
처리하기에 컴파일 타임 코드 의존성을 낮출 수 있습니다. 
   
### ECS 구성 요소
   
- Entity
   
데이터, 로직을 포함하지 않는 고유 식별자(Integer ID)입니다. 
RDBMS(Relational DataBase Management System)의 Primary Key와 같은 역할을 수행합니다.
   
- Component
   
순수 데이터(Plain Old Data, POD)의 집합입니다. 
메서드(로직)가 포함되지 않으며 C언어 Structure와 동일합니다.
   
- System
   
함수(순수 로직)입니다. 특정 컴포넌트 조합을 가진 엔티티들을 순회, 상태를 변경합니다.
   
### ECS 아키텍처 장점
   
- 메모리 지역성, 캐시 적중률 상승
   
SoA(Structure of Arrays) 방식으로 메모리에 연속 할당하는 방식을 사용합니다.
   
TransformComponent가 전체 Entity위치 업데이트 시, 메모리에 연속적으로 배열된 TransformComponent 배열만 순차적으로 읽습니다.
   
Hardware Prefetcher가 다음 메모리 주소를 예측, CPU 캐시에 미리 로드하기에 캐시 미스가 상당히 많이 줄어듭니다.
   
- Multi threading 지원
   
시스템은 서로 독립적인 컴포넌트 배열에 접근합니다. 
   
PhysicsSystem이 Transform을 읽거나 쓰고 RenderSystem이 Mesh를 읽는다면 
컴파일 타임이나 프레임 시작 지점에 Data Race가 발생하지 않도록 보장할 수 있기에 Lock 없는 
병렬 처리가 가능합니다.
   
- SIMD 명령어 활용
   
메모리에 연속배열로 데이터를 저장하기에 CPU 벡터 명령어셋을 활용한 단일 명령어 다중 데이터(SIMD) 처리를 적용, 
연산 처리량을 올릴 수 있습니다.
   
- 상속 대신 합성의 사용(Composition over Inheritance)
   
ECS에서는 엔티티에 새로운 컴포넌트를 부착하거나 제거하는 것만으로 기능정의가 가능합니다.
   
비행 몬스터를 만들고 싶다면 기존 몬스터 엔티티에 FlyComponent를 추가하기만 하면 됩니다.
   
이를 통해 좀 더 개발에 유연함을 넣을 수 있습니다.
   
- 데이터와 로직의 분리
   
시스템쪽에서 엔티티의 구체적 타입이나 전체 상태를 알 필요 없이 확인할 컴포넌트만 보유한 엔티티를 필터링, 처리합니다. 
   
모듈 간 의존성 최소화 및 단위 테스트, 디버깅이 좀 더 편해집니다.
   
- 직렬화(Serialization) 및 네트워크 동기화
   
ECS의 컴포넌트는 포인터 미포함 순수 데이터 구조체(POD)인 경우가 많아서 
게임 상태 저장, 로드 시 객체 그래프, 복잡한 객체 그래프 순회 없이 메모리 블록의 전체 복사(memcpy) 
만으로 빠른 직렬화를 할 수 있습니다.
   
또한 네트워크 게임에서 클라이언트 간 상태를 동기화할 때 변경된 특정 컴포넌트 배열만 비트 단위로 압축하여 전송이 됩니다.
   
데이터와 로직이 엄격하게 분리되기에 동일 입력 값에 대해 동일 결과를 보장하는 결정론적 시뮬레이션 구현에 유리합니다.
   
이는 리플레이 시스템, 롤백 네트워킹 구현의 기반입니다.
   
- 아키타입(Archytype) 시스템
   
내부적으로 동일한 컴포넌트 구성을 가진 엔티티들을 아키타입으로 그룹화, 관리하는 형태입니다.
   
특정 컴포넌트 조합을 가진 엔티티를 찾을 때 해당 아키타입에 속한 청크(Chunk)만 순회하면 됩니다.
   
런타임 검색 비용이 O(1)에 가깝게 유지, 많은 엔티티 처리가 가능합니다.

### ECS 아키텍처 단점
   
4가지 정도의 케이스에서 단점을 볼 수 있습니다.
   
1) Structural Change Overhead, Pipeline Stall
   
동일 컴포넌트 조합을 가진 엔티티들을 하나의 메모리 청크에 연속 할당하는 형태는
런타임 중 엔티티에 새 컴포넌트 추가, 제거와도 같은 구조적 변경에서 문제가 발생할 수 있습니다.
   
특정 엔티티 상태 변경으로 컴포넌트 조합(Signature)이 달라질 때 해당 엔티티의 모든 데이터는 
기존 메모리 청크에서 새로운 아키타입에 해당하는 메모리청크로 물리적 이동(memcpy or memmove)이 
필요하게 됩니다. 동적 메모리 재할당, 복사가 일어나는데 오버헤드가 큽니다. 
멀티 스레딩 환경에서 시스템들이 병렬로 컴포넌트 순회 중 구조적 변경 발생 시 Data Race가 생겨서 
변경 작업의 즉시 수행이 안되고 커맨드 버퍼(Command Buffer)에 쌓아뒀다가 
모든 병렬 연산이 종료되는 동기화 지점(Sync Point)에서 일괄 처리하게 됩니다. 
   
프레임 지연과 파이프라인 스톨이 발생하게 됩니다.
   
2) 계층적 데이터(Hierarchical Data), 공간 분할 트리 순회 비효율 문제
   
게임 클라에서 데이터 구조는 트리 형태의 깊은 계층 구조를 가지는게 일반적인데 
ECS 데이터 아키텍처는 1차원 선형 배열로 평탄화해서 쓰기에 보편적으로는 둘이 서로 맞지 않습니다.
   
자식 엔티티가 부모 엔티티의 Transform 데이터 참조를 하려면 부모의 고유 ID를 통해 
컴포넌트 배열에서 인덱스 룩업(Index Lookup)을 수행해야 합니다. 
   
순차적인 메모리 접근(Sequential Memory Access)가 안되고 랜덤 메모리 접근(Random Memory Access)가 
강제되면서 L1/L2 캐시 미스가 크게 납니다. 완화를 위해서 부모와 자식 데이터를 강제로 
인접하게 정렬하는 알고리즘을 프레임마다 쓰긴하는데 정렬연산을 쓰는거 자체가 코스트가 커서 딜레마가 됩니다.
   
3) 아키타입 단편화(Archetype Fragmentation) 메모리 낭비문제
   
게임 로직이 복잡해져서 많은 고유 상태(상태 이상, 임시 버프, 특수 기믹 등)가 컴포넌트로 분리되는 경우 
시스템 내에는 수만가지의 컴포넌트 조합(Archetype)이 생길 수 있습니다. 
   
특정 컴포넌트 조합을 가진 엔티티가 하나인 경우 ECS 메모리 관리자는 엔티티 하나 쓰자고 
16KB나 64KB 메모리 청크를 할당하게 됩니다. 메모리의 대부분이 비어있는 내부 단편화(Internal Fragmentation)가 발생합니다. 
시스템 순회 시 캐시 효율성이 박살나서 절차지향 프로그래밍보다 못하게 됩니다.
   
4) 이벤트 기반 통신(Event Driven Communication) 부재 문제
   
ECS는 엔티티 간 직접적 상호작용, 즉각 콜백 처리에 취약한 형태입니다.
   
OOP에서는 A객체가 B객체의 메서드를 즉시 호출(B->TakeDamage())할 수 있는데 순수 ECS에서는 이런 
직접 호출이 금지되어있습니다. 
   
A는 B에게 데미지를 주었다는 이벤트 컴포넌트를 생성, 다음 프레임이나 다음 시스템 페이즈(Phase)에서 
DamageSystem이 이 컴포넌트를 읽어서 일괄처리해야 합니다. 
   
로직이 직관적이지 않게 되고 단일 보스몬스터의 복잡한 상태 머신(State Machine)이나 UI 이벤트 트리거 
등을 구현할 경우 불필요한 보일러플레이트(Boilerplate) 코드를 양산하게 됩니다. 
   
### 언리얼 기준 ECS 도입 정도
   
언리얼 엔진은 객체지향 쪽이 큰데 Mass Entity라는 자체 ECS 프레임 워크가 제한적으로 도입되었습니다.
   
게임의 모든 개체는 UObject를 최상위 부모로 상속받으며 화면에 렌더링되고 로직을 가지는 모든 엔티티는 
AActor 클래스, UActorComponent 모델을 사용합니다. 언리얼 엔진의 가비지 컬렉션, 리플렉션(Reflection) 시스템 
블루 프린트 가상 머신, 네트워킹(Replication) 등 주요 시스템은 OOP 메모리 모델 위에서 동작하는 구조입니다.
   
제한적으로 도입된 Mass Entity는 
메트릭스 어웨이큰스 데모의 도시 트래픽 및 보행자 연산 처리같은 대규모 군중 시뮬레이션 처리가 주 목적입니다.
   
Mass Entity는 FMassEntityManager를 통해서 컴포넌트를 FMassFragment 단위로 SoA 배열로 관리하며 
캐시 일관성을 유지하는 형태입니다. 플레이어 캐릭터, 주요 보스, UI, 트리거 볼륨 등 세부 상태 관리가 
필요한 요소들은 기존 AActor 기반 OOP 아키텍처로 구현됩니다. 
ECS는 언리어에서 수만 개의 단순 객체를 병렬 처리하기 위해서 그 때만 사용하는 가속기에 해당하고 
엔진 메인 프레임워크 대체를 한 건 아닙니다. 
   
### ECS 도입 게임과 그 성과
   
- 오버워치(Overwatch, 2016)
   
블리자드의 프로메테우스 엔진(Prometheus Engine)에 ECS 아키텍처가 도입되었고 그를 기반으로 개발되었습니다.
   
ECS 적용을 통한 기술적 성과는 3가지 정도를 볼 수 있습니다.
   
1) 결정론적 넷코드(Deterministic Netcode)와 예측/롤백(Prediction & Rollback)
   
FPS에서는 클라 동기화가 정말 중요한데 ECS는 데이터와 시스템이 온전하게 분리되어있어서 
게임의 전체 상태(특정 틱의 모든 컴포넌트 데이터 배열 스냅샷)를 통으로 복사하거나 이전 프레임으로 
롤백하는 연산이 메모리 복사(memcpy)수준으로 빠르고 단순해졌습니다. 
지연 시간(Ping)을 보정하는 네트워크 최적화를 이로 달성하였습니다.
   
2) 영웅 디자인 유연성
   
여러 영웅들은 각자 다른 메커니즘을 가지는데 단일 상속 트리로 묶을 수가 없습니다.
   
많은 마이크로 컴포넌트(ex. HoverComponent, ProjectileDeflectionComponent)로 나누고 
런타임에 이를 엔티티(영웅)에 탈부착 하는 방식으로 많은 상태 이상 기믹과 상호 작용을 구현하였습니다.
   
3) 틱레이트(Tickrate) 방어
   
60Hz 이상의 서버 연산 수행을 위해서 서버 측에서도 캐시 친화적 ECS 메모리 구조를 썼고 
이를 통해 연산 병목 제거, 물리 투사체 지연 제거 및 처리를 구현하였습니다.
