# OOP의 특징에 관하여(Virtual_Inheritance, Polymorphism, Abstraction)
***
   
## 가상 상속(Virtual Inheritance) C++에서의 구현 및 다형성과 추상화
   
이전의 내용과 중복이 있을 수 있습니다. (가상화, 가상, 순수 가상 함수 등..)
   
먼저 이전의 C언어 구현을 C++로 구현한다고 할 때의 코드입니다. 
   
```C++

// 최상위 상태 클래스
#include <iostream>

class BaseState{
public:
	int health;
	BaseState() : health(100) {}

	// 순수 가상 함수 : 하위 클래스에서 반드시 구현 강제, 추상화
	virtual void showState() = 0;
	virtual ~BaseState() {} // 다형성, 가상 소멸자
};

// virtual 상속을 통한 데이터 중복 및 모호성 해결
class Derived1 : virtual public BaseState{
public:
	int attack;
	Derived1() : attack(10) {}
};

class Derived2 : virtual public BaseState{
public:
	int defence;
	derived2() : defence(20) {}
};

// 다중 상속 구현 클래스
class DiamondObject : public Derived1, public Derived2{
public:
	// 부모의 순수 가상 함수 구체화(오버라이딩), 다형성 구현
	void showState() override{
		std::cout << "Health: " << health
				  << ", Atk : " << attack
				  << ", Def : " << defecnce << "\n";
	}
};

int main(){
	DiamondObject obj;

	// 가상 상속으로 base_state를 하나로 통합, 모호성 해결로 접근 문제 해결
	obj.health = 150;

	// 다형성 이용, 객체 접근
	BaseState* ptr = &obj;
	ptr->showState(); // DiamondObject의 showState() 호출

	return 0;
}
```

이전의 C언어에서는 C언어 기반의 상속 구현에 있어서 obj.d1.base.health, obj.d2.base.health로
데이터 불일치 및 메모리 낭비가 발생하게 됩니다. 이를 C++에서 가상 상속을 통해서 해결합니다. 
해당 분리의 구현은 virtual public BaseState 가상 상속입니다. 
   
virtual public BaseState 사용으로 C++상에서는 컴파일러가 내부적으로 포인터(vbptr)를 조정, 
BaseState, 최상위 부모의 인스턴스가 메모리에 하나만 존재하도록 보장합니다. 
   
결과적으로 obj.health = 150; 과 같은 형태가 되며 이는 직관적이고, 모호성이 없습니다. 
   
위의 코드에서 BaseState 클래스 내에 virtual void showState() = 0;의 순수 가상 함수로 추상화를 
구현합니다. 
   
BaseState는 health를 가진 기본 상태라는 데이터 구조를 제공하며 showState의 구현은 없습니다. 
해당 클래스의 하위 클래스에서는 attack든 defence든 둘 다 가지든 뭐가 구현될지 모릅니다. 
순수 가상 함수를 통한 추상화로 하위 클래스가 따로 가지는 그 무언가를 같이 묶어서 구현해야 한다는 규약입니다. 
   
>클라이언트 상에서의 가상 함수, 가상 함수를 사용함으로써 얻는 이득
   
화면에 여러 객체를 그려야 합니다. 내가 움직이는 플레이어블 캐릭터도 그려야 하고 투사체도 그려야 하고 
몬스터도 그려야 하고 뭐 이거저거 게임하면서 보이는 그거들 다 그려야 합니다. 
   
모두 GameObject라는 부모 클래스를 상속받는다고 가정합니다. 가상 함수가 없을 경우 메인 렌더링 루프에서 
쭉 보면서 객체가 몬스터인지 투사체인지 캐릭터인지 뭔지 다 타입 검사하고 그 검사하고 나서 각자 렌더링 함수를 
호출해야합니다. 한 번 하면 되는 일을 두 세번 하는 것 같이 보입니다. 
   
GameObject내 virtual void Render()라는 가상 함수를 하나 둔다고 가정합니다. 
   
메인 루프에서 GameObject* 포인터들을 모아둔 배열을 순회, ->Render()를 일괄 호출하는 방식으로 진행합니다. 
가상 함수 메커니즘으로 각자 타입에 맞게 알아서 렌더링 코드를 실행합니다. 
   
이는 컴파일러가 메모리를 알아서 엮어준 것 때문인데 다음의 두 가지가 주요 구조입니다.
   
1) vtable(가상 함수 테이블)
   
클래스 단위로 생성되는 배열입니다. 안보입니다. 해당 클래스가 실제로 호출해야 할 
가상 함수들의 메모리 주소가 순서대로 기록되어 있습니다. 
   
가상 함수 테이블은 프로그램 읽기 전용 데이터에 하나만 만들어집니다.
   
2) vptr(가상 포인터)
   
객체(인스턴스)가 생성될 때마다, 컴파일러가 객체의 메모리 레이아웃 맨 앞에 집어넣는 포인터입니다. 
해당 포인터는 자신이 속한 클래스의 vtable을 가리키도록 초기화됩니다. 
   
실행 시간에 부모 포인터로 가상 함수 호출 시 여러 단계의 간접 참조(Indirect Call)형태의 과정이 발생합니다. 
해당 과정은 다음과 같습니다.
   
- 객체 내부 접근 -> vptr 읽기 -> vtable 찾아가기 -> 실제 함수 주소 꺼내오기 -> 해당 주소로 점프
   
다시 BaseState로 돌아가서
일반 가상 함수 virtual void showState() = {};, 빈 함수로 구현되는 경우에는 하위 클래스 
DiamondObject가 실수로 showState() 구현을 하지 않아도 에러가 발생하지 않습니다. 
   
순수 가상 함수 virtual void showState() = 0; 로 구현되는 경우 하위 클래스 
DiamondObject가 showState() 구현을 하지 않고 진행할 시 에러가 발생합니다. 
   
해당 클래스는 미완성 클래스로 취급되며 컴파일 에러가 발생합니다. 
   
추상화 구현 과정에서의 VTable(가상 함수 테이블) 값의 변화 과정을 통해 객체가 나오기까지의 과정을 알 수 있습니다. 
   
처음에 BaseState의 VTable내 showState가 들어가야 할 공간은 비어있거나 호출되는 경우 에러 발생되는 
더미 핸들러 주소를 가리키고 있습니다. 미완성 테이블을 가진 BaseState로는 객체 생성이 불가능합니다. 
추상 클래스는 이렇습니다. 인스턴스 생성이 안됩니다. 
   
하위 클래스인 DiamondObject가 attack, defence를 활용, void showState() override {...}를 구현하게 되면 
DiamondObject의 VTable을 만들 때 앞의 showState 공간에 구현된 새 함수의 메모리 주소를 채워 넣습니다. 
이런 식으로 추상 클래스의 하위 클래스가 구현됩니다. 이 클래스는 객체(인스턴스)를 생성할 수 있습니다. 
   
추상 클래스는 전체 틀을 잡아주는 형태입니다. 그를 상속받는 모든 하위 클래스 객체들을 하나의 타입(포인터 배열, 벡터 등..)으로 
묶어서 관리할 수 있습니다. 
   
다형성입니다. 
   
위 코드 기준으로 다형성이 나타나는 곳은
   
```C++
BaseState* ptr = &obj;
ptr->showState(); // DiamondObject의 showState()를 호출합니다.
```
   
이 부분입니다.
   
다형성의 정의입니다.
   
컴파일 타임(Complie Time)에 호출자는 상위 클래스의 정적 타입(Static Type)포인터를 통해 
가상 함수 호출 인스트럭션을 생성하나, 런타임(Run Time)에는 객체 메모리 최상단에 위치한 
가상 포인터(vptr)가 가리키는 가상 함수 테이블(vtable)을 참조하는 동적 바인딩(Dynamic Binding)이 
발생하여 실제 메모리에 할당된 파생 클래스(Dynamic Type)의 오버라이딩된 함수 주소로 
간접 분기(Indirect Call)되어 실행되는 아키텍처 수준의 메커니즘 
   
위의 코드에서 ptr의 타입은 BaseState로 인식되는데 인식 시점에 showState()가 실제 메모리 어디에 있는 지 
알 수 없습니다. 특정 주소로 바로 점프하라는 어셈블리 명령어(Direct Call)를 만들지 못합니다. 
   
이 대신 나중에 런타임이 되면 ptr이 가리키는 메모리 주소로 가서 해당 주소의 함수를 실행하라는 
조건부 호출 명령만 기계어로 바꿔둡니다. 
   
프로그램 런타임 시점에 DiamondObject 타입의 obj가 메모리에 할당 시, 객체의 메모리 레이아웃 
가장 앞단의 오프셋 0에는 vptr이라는 64비트 기준 8바이트의 포인터가 들어갑니다. 
   
이 vptr는 .rotate 영역의 데이터 세그먼트에 존재하는 DiamondObject 전용의 vtable 주소를 가리키고 있게 됩니다. 
호출자의 지시가 내려왔을 때, CPU는 ptr이 가리키는 주소로 이동, vptr을 읽고 그 값으로 해당 객체의 
vtable로 이동합니다. 
   
vtable 내에는 DiamondObject가 override 키워드를 통해 구현한 showState() 함수의 메모리 주소가 
기록되어 있으며 이 vtable에서 함수 주소를 CPU가, CPU가 이 vtable 내에서 함수 주소를 레지스터로 
읽어온 뒤 그 주소로 점프(Indirect Branch/Call)하여 기계어를 실행합니다. 
BaseState의 빈 공간이 아닌 동적 타입인 파생 클래스의 고유 메모리 주소로 치환하여 실행됩니다. 
이것이 다형성의 일련의 과정입니다.
