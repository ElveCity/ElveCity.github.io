# 컴파일러 최적화(Compiler Optimizaion)
***
   
## 최적화 레벨(-00 ~ -03)
   
컴파일러 최적화는 총 4단계로 나뉘며 -00 레벨부터 -03 레벨까지이고 -03으로 갈수록 공격적인 최적화를 진행합니다.
   
### -00 레벨(최적화 없음)
   
개발, 디버그 빌드 표준입니다. 가장 낮은 레벨이며 컴파일 시간이 짧습니다. 
   
소스 코드 제어 흐름과 변수 상태가 컴파일 이후에도 1:1 매칭되는 특징이 있습니다. 보기 편합니다.
   
해당 C++ 예시는 다음과 같습니다.
   
```C++
int add(int a, int b){
	int result = a + b;
	return result;
}
```
   
해당 레벨에선 내부 파이프라인으로 볼 때 
프론트엔드에서 생성한 AST(Abstract Syntax Tree)를 IR(Intermediate Representation)로 변환, 
최적화 패스(Optimization Passes)를 거의 거치지 않고 바로 백엔드로 넘기고 기계어로 만듭니다. 
   
레지스터 할당기(Register Allocator)는 변수를 레지스터에 캐싱하지 않고 매번 스택(메모리)에 할당하게 됩니다. 

따라서 변수 접근이 스택을 통해 이뤄지며 예시로는
   
```
mov DWORD PTR [rbp-4], edi // 인자 메모리 저장

mov eax, DWORD PTR [rbp-4] // 메모리에서 레지스터로 로드 후 연산
```
   
이 있고 명령어 사이에 불필요한 메모리 I/O가 지속적으로 발생하게 됩니다.
   
-00레벨의 장점은 컴파일 속도가 빠르고 디버그 시 보기 좋다는겁니다. GDB(GNU Debugger) 등의 디버거 
사용 시, 코드 라인 스텝핑과 변수 값 조회가 소스코드와 잘 일치합니다. 
   
대신 런타임 실행 속도가 느립니다. 메모리 접근 오버헤드도 큰 단점이 있습니다. 
   
### -01 레벨(기본 최적화)
   
-01 레벨 최적화에서는 컴파일 시간은 크게 안늘어나면서 실행 파일 크기 줄이기, 속도 높이기를 하려고 씁니다. 
   
해당 C++ 예시는 다음과 같습니다. 
   
```C++
int calculate(){
	int x = 10;
	int y = 20;
	int z = x * y; // 상수 폴딩 대상

	return z;
}
```
   
내부 파이프라인으로 볼 때 
IR 단계에서 Mem2Reg(메모리 접근을 레지스터 접근으로 전환), 상수 폴딩(Constant Folding), 
데드 코드 제거(Dead Code Elimination) 등의 기본적인 최적화 패스를 씁니다. 
   
-01 레벨 최적화에서는 변수들이 스택(메모리) 대신 레지스터에 할당됩니다.
   
위 코드 기준에서는 어셈블리단에서 런타임 곱셈 명령(imul)이 없어지고 200이라는 
상수를 반환하라는 명령(mov eax, 200)으로 치환됩니다.
   
해당 레벨의 최적화 사용 시의 장점으로는 코드 크기 감소, 기본 런타임 성능의 확보입니다.
   
대신 디버깅 시 일부 변수가 최적화로 없어집니다.(optimized out) 값 추적이 안됩니다.
   
그래도 안전한 최적화만 해서 리스크는 적은 편입니다. 디버깅하면 코드 흐름이 좀 건너뛰는 것처럼 
보일 수 있습니다.
   
### -02 레벨(권장/표준 최적화)
   
-02 레벨 최적화에서는 루프 언롤링 빼고 최적화 기능 거의 다 씁니다. 
코드 크기를 너무 크게 늘리지 않는 선에서 대부분 애플리케이션(릴리즈 빌드)에서 표준으로 씁니다.
   
해당 C++ 예시는 다음과 같습니다.
   
```C++
void process(int* __restrict a, int* __restrict b, int val){
	for (int i = 0, i < 1000; ++I){
		a[i] = b[i] + val;
	}
}
```
   
내부 파이프라인으로 볼 때
명령어 스케줄링(Instruction Scheduling), 공통 부문식 제거(Common Subexpression Elimination), 
엄격한 에일리어싱(Strict Aliasing)기반 포인터 최적화, 분기 예측(Branch Prediction) 최적화가 
추가됩니다. 
   
-02 레벨 최적화에서는 CPU 파이프라인 스톨 방지를 위해 명령어의 순서가 재배치(Instruction Reordering) 
됩니다. 이 재배치에서 소스 코드의 순서와 명령어의 순서는 다르게 됩니다. 
   
반복문 내에서 변하지 않는 val 로드 같은 연산은 루프 밖으로 이동(Loop Invariant Motion)시킵니다. 
포인터 간 메모리가 겹치지 않는다고 가정하여 캐시효율을 최대한 높이는 레지스터 로드가 수행됩니다.
   
해당 레벨 최적화 이점은 컴파일된 바이너리 실행 속도의 큰 폭 증가입니다. 
성능과 바이너리 크기 간 좋은 벨런스 유지가 됩니다.
   
대신 컴파일 시간이 길고 디버깅이 안됩니다. 그냥 안됩니다.
   
float*를 int\*로 캐스팅, 접근과 같은 코드는 -00레벨에서는 정상적으로 동작하다가 -02레벨에서 
Undefined Behavior가 나면서 크래시가 발생할 수 있습니다. 컴파일러 최적화 정도 때문도 있지만 
해당 캐스팅, 접근은 Strict Aliasing Rule 위반에 해당해서 그렇습니다.
   
### -03 레벨(공격적 최적화)
   
마지막으로 -03 레벨 최적화입니다. 극단적인 성능중시입니다. 
   
바이너리의 크기나 컴파일 시간 다 신경 안씁니다. 가능한 모든 연산 자원을 쓰고 최대 성능을 뽑기 위해섭니다. 
수학 연산이 극도로 많은 엔진 코어, 고성능 컴퓨팅에 이용되는 최적화 방식입니다.
   
기본적으로 단일 번역 단위(Translation Unit, 개별 .cpp 파일)내에서만 이루어집니다. 
   
해당 C++ 예시는 다음과 같습니다.
   
```C++
void add_arrays(float* res, const float* a, const float* b, int size){
	for (int i = 0; i < size; ++I){
		res[i] = a[i] + b[i];
	}
}
```
   
내부 파이프라인으로 볼 때 
-02의 모든 패스에 더해, 공격적 루프 언롤링(Loop Unrolling), 자동 벡터화(Auto-Vectorization, SIMD), 
공격적 함수 인라이닝(Aggressive Inlining)이 IR 패스에 추가 적용됩니다.
   
어셈블리 단에서 일반적인 addss 명령어가 아닌 vaddps(AVX 명령어 등)가 생성되어 한 번의 CPU 사이클에 
4개 이상의 요소를 병렬로 처리하게 됩니다. 단일 명령어 다중 데이터(SIMD) 명령어가 많이 사용됩니다. 
   
분기 패널티를 줄이기 위해 반복문 내부 코드가 여러번 복사되어 나열(Loop Unrolling)됩니다.
   
-03 레벨 최적화의 장점은 행렬 연산, 대규모 배열 처리 등에서 하드웨어의 이론상 최대 처리량 달성을 
볼 수 있습니다. 극단적인 성능을 볼 수 있습니다. 
   
그렇게 극단적으로 쓰는만큼 코드 블로트(Code Bloat, 바이너리 크기 비대화)가 발생하는 단점이 있습니다. 
인라이닝과 루프 언롤링으로 인해 기계어 양이 엄청 많이 나옵니다.
   
이렇게 바이너리 크기가 엄청 커지면서 CPU의 L1 명령어 캐시 용량초과가 발생 시 캐시 미스가 너무 많아져서 
-02 레벨 최적화보다 런타임 성능이 낮아질 수도 있습니다.
   
#### 루프 언롤링(Loop Unrolling)
   
루프 언롤링은 반복문이 실행될 때 발생하는 오버헤드를 줄이고, 
CPU의 명령어 수준 병렬성(ILP, Instruction Level Parallelism)을 높이기 위해 코드를 펼치는 방법입니다. 
   
다음은 최적화 미적용 기본 반복문 예시입니다.
   
```C++
void scale_array(float* arr){
	for (int i = 0; i < 400; ++i){
		arr[i] = arr[i] * 2.0f;
	}
}
```
   
위의 코드의 루프 제어 오버헤드(Loop Control Overhead)는 다음의 3가지가 발생합니다.
   
1) 조건 검사
   
i < 400 이 참인지 비교(cmp 명령어)
   
2) 분기(Branch)
   
조건이 참이면 루프 시작점으로 점프(jl or jne 명령어)
   
3) 인덱스 증가
   
++i 실행(inc or odd 명령어)
   
실제 필요 연산은 배열 요소에 2.0을 곱하는 건데 곱셈 한 번에 부가 제어 명령어가 3개 들어갑니다.
   
400번의 반복에서 제어 오버헤드도 400번 나옵니다.
   
다음은 루프 언롤링(Factor = 4 기준) 적용 반복문 예시입니다.
   
```C++
void scale_array_unrolled(float* arr){
	int i = 0;
	//
	for (; i < 400; i += 4){
		arr[i]		= arr[i]		* 2.0f;
		arr[i + 1]	= arr[i + 1]	* 2.0f;
		arr[i + 2]	= arr[i + 2]	* 2.0f;
		arr[i + 3]	= arr[i + 3]	* 2.0f;
	}

	for (; i < 400; ++I){
		arr[i] = arr[i] * 2.0f;
	}
}
```
   
위 코드는 컴파일러 -03레벨 최적화 분석이나 수동 루프 언롤링 적용의 예시입니다.
   
이렇게 언롤링하는 경우 다음의 두 가지 장점을 볼 수 있습니다.

1) 제어 명령어 감소
   
루프가 400번 도는 대신 100번만 돌아서 조건검사와 분기 명령어 실행 횟수가 75퍼 줄어들고 CPU 사이클이 절약됩니다.
   
2) 분기 예측(Branch Prediction) 부담 완화
   
분기문이 적어지므로 CPU의 분기 예측 실패(Branch Misprediction) 확률이 낮아집니다.
   
분기 예측 실패 시 발생하는 파이프라인 플러시(Pipeline Flush)비용이 너무 커서 
방지하면 좋습니다.
   
루프 언롤링을 통해 데이터 의존성 제거 및 명령어 수준 병렬성(ILP, Instruction Level Parallelism) 
의 향상을 볼 수 있습니다.
   
다음의 누산(Accumulation)예시를 통해 장점을 볼 수 있습니다.
   
```C++
float sum_array(const float* arr){
	float sum = 0.0f;
	for (int i = 0; i< 100; ++i){
		sum += arr[i];
		// Read After Write (RAM) 데이터 의존성 발생
	}
	return sum;
}
```
   
위 코드는 이전 루프의 덧셈 결과가 sum에 기록되어야 다음 루프 덧셈이 가능해집니다. 
   
슈퍼스칼라(Superscalar) CPU는 한 버넹 여러 명령어를 병렬로 처리 가능하나 이 의존성으로 인해 
파이프라인 스톨이 발생, 산술논리연산장치(ALU, Arithmetic Logic Unit)가 쉬는 구간이 생깁니다.
   
다음은 언롤링 및 누산기 분리 적용 예시입니다
   
```C++
float sum_array_unrolled(const float* arr){
	float sum0 = 0.0f;
	float sum1 = 0.0f;
	float sum2 = 0.0f;
	float sum3 = 0.0f;

	for (int i = 0; i < 100; i+= 4){
		sum0 += arr[i];
		sum1 += arr[i + 1];
		sum2 += arr[i + 2];
		sum3 += arr[i + 3];
	}

	return sum0 + sum1 + sum2 + sum3;
}
```
   
위 코드에서 sum0부터 sum3까지 서로 데이터 의존성이 없습니다. 
따라서 CPU는 이 4개의 덧셈 명령어를 각기 다른 ALU나 FPU(부동 소수점 유닛, Floating Point Unit)에 
할당, 동시에 연산이 가능하게 됩니다. 
   
컴파일러가 -03 레벨 최적화에서 루프 언롤링으로 레지스터 이름을 재할당(Register Renaming)하는 이유입니다.
   
언롤링된 루프는 컴파일러에서 SIMD(Single Instruction Multiple Data)명령어로 압축(Vectorization)합니다.
개별 스칼라 연산으로 그대로 두지 않습니다.
   
위 scale_array_unrolled 코드 루프 바디는 어셈블리단에서 AVX/SSE Instruction set으로 
하나의 명령어로 치환됩니다. 스칼라 상태에서는 8번 반복되는 연산 명령어가 1개 명령어로 압축됩니다. 
   
다음의 3가지 어셈블리 명령어가 순차적으로 파이프라인에 입력되게 됩니다.
   
```
vmovups ymm0, [rax]
```
   
메모리 로드입니다.
   
256비트(float 8개)를 한 번에 레지스터로 로드합니다.
위 코드에서는 언롤링 펙터(Unrolling Factor)가 4개 기준으로 작성된 예시이며 4개 들어가 있어서
4개의 float입니다.
   
원래는 4개의(최대 스펙 기준 8개) movvss가 1개의 vmovups로 치환되는 형태입니다.
   
```
vmulps ymm0, ymm0, ymm1
```
   
벡터 곱셈입니다. 로드된 4개의(최대 스펙 기준 8개) float에 동시에 2.0f을 곱합니다.
   
4개의 mulss가 1개의 vmulps로 치환되는 형태입니다.
   
```
vmovups [rax], ymm0
```
   
메모리 스토어입니다. 결과를 다시 메모리에 저장합니다.
   
4개의 movss가 1개의 vmovups로 치환되는 형태입니다.
   
이런 치환으로 줄인 만큼 n배 빨라지는 형태입니다 8개를 1개로 줄이면 8배 빨라지고 그렇습니다. 
   
루프 언롤링 너무 막써서 기계어 너무 커지고 그게 L1 명령어 캐시 안에 다 못드가면 
명령어 캐시 미스 막 나옵니다. 파이프라인 병렬화로 이득보고 메모리 지연으로 다 손해봅니다.
L2 캐시 가져다 쓰는 형태여도 충분히 느려져서 손해입니다. 잘 보고 해야합니다.
